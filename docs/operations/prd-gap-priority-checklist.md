# PRD 差异修复优先级清单

本清单根据 [Vivi-Drop-产品需求文档.md](/Volumes/T7/Dev/Web/SyncFlow/Vivi-Drop-产品需求文档.md) 与当前代码实现的差异整理，目标是先收敛“用户感知最强、最影响验收”的缺口。

说明：

- 这里是“严格按 PRD 判定”的差异清单
- 不代表每一项都必须立即改；优先级以用户影响、可解释性、验收风险排序
- 当前代码已经实现的主流程（手动优先、不打断当前 TCP 文件、分页状态刷新）不再重复列入

## P0

### 1. 统一用户提示通道

目标：

- 把上传相关的关键失败 / 中断 / 离线提示，统一成用户可见反馈
- 不再依赖 `console.warn` 或只有状态变化但没有提示

当前差异：

- `SyncActivityScreen` 只对暂停 / 恢复 / 取消失败做了 `Alert`
- `AlbumWorkbenchScreen` 对自动上传配置保存失败只有 `console.warn`
- native 已有 `onError` 事件，但首页和相册页没有统一消费

建议动作：

1. 明确移动端统一提示机制：先用 `Alert` 还是补一个轻量 Toast 组件
2. 让 `SyncActivityScreen` / `AlbumWorkbenchScreen` 订阅并消费 `onError`
3. 覆盖以下提示：
   - 设备已断开，上传已暂停
   - 自动上传已中断
   - 传输中断，请稍后重试
   - 自动上传开启失败，请重试
   - 暂停失败，请重试
   - 恢复失败，请重试
   - 取消失败，请重试

验收标准：

- 用户在上传中断、设备离线、配置失败时能立即看到可见提示
- 不再出现 native 发了错误事件但 UI 无感知

## P1

### 2. 自动上传配置前置校验补齐

目标：

- 让自动上传配置真正符合 PRD 的“开启前必须满足条件”

当前差异：

- 开启自动上传时，时间范围没有显式“必须选择”校验，而是依赖默认值
- `custom` 模式下，输入为空时没有阻止保存
- 保存失败时没有用户可见反馈

建议动作：

1. 明确 `enabled=false -> true` 时的前置校验顺序
2. `timeRangeMode === 'custom'` 时：
   - 输入为空禁止保存
   - 非法时间禁止保存
   - 保持当前配置不变
3. 保存失败时给出用户提示
4. 明确“此时此刻 / 今天开始 / 所有素材 / 自定义时间点”的 UI 状态与默认值语义

验收标准：

- 开启自动上传前，配置不完整时不能直接开启
- 自定义时间点不合法时不会落库
- 保存失败时用户明确知道失败

### 3. 共享文件页前置校验与异常分流

目标：

- 让共享文件页在“设备不可用 / 目录不可访问 / 普通加载失败”之间有明确区分

当前差异：

- 进入页时直接请求共享目录
- 加载失败统一落成通用错误态
- 没有先判断当前连接电脑是否可用且在线

建议动作：

1. 进入共享文件页前，先校验当前绑定设备与连接状态
2. 区分以下错误：
   - 当前设备不可用
   - 共享目录不可访问
   - 普通网络 / 请求失败
3. 保留刷新能力与空目录状态

验收标准：

- 空目录展示空状态，不视为错误
- 设备不可用与目录不可访问有不同提示
- 加载失败仍可刷新重试

## P2

### 4. 上传控制文案与按钮状态再对齐

目标：

- 进一步把首页控制按钮收敛到 PRD 表述

当前差异：

- 目前逻辑已经基本正确，但仍是工程语义主导，不是严格产品语义主导
- PRD 更强调“当前任务来源决定操作按钮”

建议动作：

1. 再做一轮状态矩阵核对：
   - 当前任务 manual + auto active
   - 当前任务 manual + auto paused
   - 当前任务 auto + auto active
   - 无当前任务 + auto paused
2. 确认按钮文案、可见性、优先级与设计预期一致
3. 确认切 tab / 回前台后不会出现按钮消失或错位

验收标准：

- 首页按钮在各状态下稳定可解释
- 不再出现“恢复自动上传”或“取消本次手动上传”异常隐藏

### 5. 重复素材提示细化

目标：

- 把“已自动跳过重复素材”从技术结果变成产品可理解提示

当前差异：

- 现在会返回 `skippedCount`
- UI 也会在提交成功后显示“跳过 X 个重复项”
- 但提示样式与 PRD 的即时反馈定义还不完全一致

建议动作：

1. 明确这条提示使用成功提示内联文案，还是独立 Toast
2. 区分：
   - 全部成功入队
   - 部分重复被跳过
   - 全部都被跳过

验收标准：

- 用户能理解哪些素材被自动去重
- 不会误以为点击上传没生效

## P3

### 6. PRD 与当前实现的语义偏差确认

目标：

- 明确哪些差异是“真的要改”，哪些是产品策略已经变化

当前差异：

- PRD 写的是“当前自动任务停止 / 当前批次停止”
- 现实现是“当前已进入 TCP 的文件传完再停”

建议动作：

1. 由产品或你这边确认：
   - 是否真的需要 mid-stream abort
   - 还是以当前“文件级原子性优先”作为新准则
2. 如果确认沿用当前实现：
   - 更新 PRD 或补一份当前行为说明

验收标准：

- 文档与实现不再互相打架
- 后续 review 不会反复卡在同一语义分歧

## 建议执行顺序

1. 先补 `P0` 的统一错误提示
2. 再补 `P1` 的自动上传配置校验
3. 然后处理共享文件页前置校验
4. 最后做 `P2/P3` 的产品语义收尾
