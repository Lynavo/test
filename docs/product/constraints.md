# SyncFlow 产品约束与非目标

本文件把当前项目的核心产品约束从工程语言翻译成产品语言，方便交接时统一判断边界。

## 1. 当前产品边界

SyncFlow V2 当前只做一件事：

- `iPhone -> Desktop（macOS / Windows）` 的局域网自动增量同步

明确不在当前范围内：

1. Linux 桌面端
2. Android 客户端
3. 云端中转/云备份
4. 用户手动挑选、删除、跳过、重排队列
5. 多文件并发上传

补充说明：

- macOS / Windows 都属于当前桌面端范围
- 不要求所有辅助能力在两个平台上完全同构；平台差异以当前实现为准，例如 Windows 的共享地址以手动配置为主

## 2. 必须保持的产品约束

### 2.1 队列只读

用户可以看到队列，但不能：

1. 删除队列项
2. 跳过队列项
3. 修改顺序
4. 手动标记已完成

原因：

- 当前产品定位是“无感自动同步”，不是“手动导出工具”

### 2.2 自动增量

用户不需要手动勾选素材。

系统应自动：

1. 扫描相册
2. 识别未同步素材
3. 自动排队
4. 自动续传和恢复

### 2.3 单文件串行

同一台手机同一时间只允许传 1 个文件。

原因：

1. 简化状态机和续传逻辑
2. 降低后台和重连场景的复杂度
3. 让历史统计和错误恢复更可解释

### 2.4 以完成为准的历史统计

历史和 dashboard 统计的关键语义是：

- “今天同步了多少”，按 **desktop sidecar 完成落盘** 计算

不是：

- 按拍摄日期
- 按素材创建日期
- 按 iPhone 本地 UTC 日期

### 2.5 renderer 隔离

desktop renderer 不直接访问：

1. SQLite
2. sidecar HTTP
3. 文件系统

这不是单纯技术洁癖，而是产品稳定性约束：

- 诊断、权限、签名、打包、sidecar 生命周期必须可控

## 3. 当前用户承诺

对用户实际承诺的是：

1. 自动发现和绑定桌面端设备
2. 自动增量同步
3. 中断后尽可能自动恢复
4. 在 desktop 端能看到进度、历史、存储位置和诊断导出

不是承诺：

1. 精细的手工传输控制
2. 云端统一素材管理
3. 多平台全覆盖

## 4. 体验优先级

当前阶段优先级排序：

1. **正确性**：不丢文件、不乱记历史、不重复写坏文件
2. **恢复能力**：断网、重启、后台后能继续
3. **可解释性**：状态提示、诊断导出、历史统计可读
4. **性能**：在上述前提下尽量快

这意味着：

- 不要为了 UI 更丝滑轻易牺牲 ACK 或传输稳定性
- 不要为了“可操作”而引入手工队列控制，把产品方向带偏

## 5. 当前高级功能的定位

### 5.1 共享地址

`共享地址（局域网）` 当前更适合定位为：

- 高级功能
- 诊断/辅助访问能力

不是主同步链路。

当前平台差异：

- macOS：可结合系统文件共享做自动检测
- Windows：以手动配置共享为主，UI 提供快速入口和推荐地址

### 5.2 诊断导出

诊断导出不是面向普通用户的卖点，而是 beta 阶段必须具备的运维工具。

## 6. 后续扩展时的判断原则

如果有人提出新需求，先问：

1. 会不会破坏“自动增量同步”的定位？
2. 会不会引入队列可写、手工控制、并发上传？
3. 会不会让历史和状态语义更难解释？
4. 会不会让 renderer 绕开 main/preload/sidecar 边界？

任一答案为“会”，就应先审慎评估，而不是直接做。
